# Слежение за папкой

## Цель

Изменения, сделанные в открытой папке не Проводником — Claude создал файл, браузер докачал архив, коллега положил документ на сетевую шару, — должны появляться в списке сами. Сейчас список перечитывается только после собственных файловых операций и при навигации, поэтому созданный извне файл виден лишь после ухода из папки и возврата.

Ограничение, определяющее весь дизайн: **обновление не имеет права стоить дорого**. Проводник существует ради скорости, и слежение, перечитывающее каталог на каждый чих, съело бы ровно ту выгоду, ради которой `DirectoryLoader` написан на `opendir`/`d_type`. Поэтому событие от файловой системы приводит не к полной перезагрузке, а к точечному диффу.

## Архитектура

### DirectoryWatcher — новый файл `Sources/PathwayCore/DirectoryWatcher.swift`

Обёртка над FSEvents для **одной** папки. Публичный интерфейс — три метода:

```swift
public protocol DirectoryWatching: AnyObject, Sendable {
    func start(_ directory: URL, onChange: @escaping @MainActor (DirectoryChange) -> Void)
    func stop()
}
```

`DirectoryChange` — что именно произошло, в объёме, который даёт FSEvents:

```swift
public struct DirectoryChange: Sendable {
    /// Событие несло флаг ItemModified — содержимое существующего объекта
    /// изменилось, и метаданные всего списка устарели.
    public let hasModifications: Bool
}
```

Одно поле, а не перечисление событий по файлам: FSEvents на сетевых томах и при коалесцировании отдаёт события пачкой и без надёжного соответствия «одно событие — один файл». Полагаться на пути из события значило бы строить дифф на данных, которых система не гарантирует. Пути в `DirectoryChange` не попадают намеренно — потребитель всё равно перечитывает каталог, а флаг модификации нужен лишь чтобы решить, дочитывать ли метаданные.

Протокол, а не конкретный тип, — по правилу проекта: каждая граница с ОС подменяема (`Mounting`, `CredentialStoring`, `ReleaseFetching`). Здесь это ещё и единственный способ протестировать реакцию модели: настоящий FSEvents в тесте потребовал бы писать в файловую систему и ждать неопределённое время.

Реализация — `FSEventStreamCreate` со следующими параметрами и обоснованием каждого:

- **Только сама папка, без рекурсии.** Путь в `pathsToWatch` один. FSEvents рекурсивен по своей природе, и отфильтровать вложенные изменения обязан потребитель: событие в подпапке на состав нашего списка не влияет. Фильтрация — сравнение `deletingLastPathComponent()` пути события с отслеживаемой папкой.
- **`latency` 0.3 с.** Коалесцирование делает сама система: пачка из тридцати созданных файлов приходит одним вызовом, а не тридцатью. Своего дебаунса в проекте поэтому не будет.
- **`kFSEventStreamCreateFlagFileEvents`.** Без него события приходят на уровне каталога и флаг `ItemModified` не различить. Он же даёт `ItemCreated`/`ItemRemoved`/`ItemRenamed`.
- **`kFSEventStreamCreateFlagNoDefer`.** Первое событие пачки доставляется сразу, а `latency` работает как окно после него. С `Defer` создание одного файла ждало бы 0.3 с впустую — а это ровно тот случай, который пользователь видит чаще всего.
- **`kFSEventStreamEventIdSinceNow`.** История изменений нам не нужна: список только что прочитан.

Поток — выделенная последовательная `DispatchQueue` через `FSEventStreamSetDispatchQueue`. Колбэк переводит вызов на `MainActor` сам, потребителю доставляется уже на главном потоке.

`stop()` обязан звать `FSEventStreamStop` → `Invalidate` → `Release` именно в этом порядке и быть идемпотентным: он вызывается и из `start` при смене папки, и из `deinit`.

### BrowserModel — подписка и дифф

Модель получает приватный `watcher: any DirectoryWatching`, инъектируемый через инициализатор (по умолчанию — настоящий `DirectoryWatcher`).

**Подписка переустанавливается в конце `reloadAsync`**, когда список уже прочитан, — не в начале и не в `navigate`. Причина: слежение за папкой, чтение которой ещё не закончилось, породило бы дифф против неполного списка. Ставить в `navigate` нельзя ещё и потому, что мимо него идут `goBack`/`goForward`/`goUp` и первая загрузка вкладки.

Новый приватный метод `applyExternalChange(_:)` — реакция на событие:

```
1. Открыт инлайн-редактор имени → выходим, ничего не делая.
2. Идёт своя файловая операция (isBusy) → выходим.
3. Читаем имена (Task.detached, .utility).
4. Барьер: pane.path не сменился, задача не отменена.
5. Дифф против items по URL:
   — новые записи берём из свежего списка (metadataLoaded == false);
   — исчезнувшие выбрасываем;
   — уцелевшие оставляем как есть, с уже прочитанными метаданными.
6. Если change.hasModifications — у всех уцелевших сбрасываем metadataLoaded.
7. Чистим выделение от исчезнувших URL.
8. items = sorted(merged), кладём в кэш.
9. Дочитываем метаданные (Task.detached, .utility) для записей с
   metadataLoaded == false и обновляем список вторым присваиванием.
```

Шаг 6 — это ответ на случай «файл дописан, имя прежнее»: состав списка не изменился, но размер и дата в колонках устарели. Сбрасывать флаг всегда нельзя — тогда каждое событие тянуло бы `stat` по всей папке, и дифф потерял бы смысл.

Шаги 1 и 2 — защита от обновления, которое пользователь воспримет как сбой. Инлайн-редактор при `reloadData` теряет фокус, и имя, которое человек набирал, пропадает. Своя операция сама зовёт `reload()` в `run(_:)`, и второе обновление поверх неё — лишняя работа с гонкой в придачу.

Приоритет `.utility`, а не `.userInitiated`: обновление фоновое, пользователь его не ждал, и оно не должно конкурировать с загрузкой папки, в которую он только что перешёл.

**`errorMessage` при внешнем обновлении не выставляется.** Ошибка чтения здесь означает, что папку удалили или отмонтировали, — алерт поверх работы в другой вкладке был бы неуместен; список просто остаётся прежним. Это отличает `applyExternalChange` от `reload()`, где ошибку показывать обязательно: там чтение запросил человек.

Отдельная `Task` для внешнего обновления (`refreshTask`) — не переиспользуем `loadTask`: отмена загрузки при закрытии вкладки не должна зависеть от того, пришло ли событие от ФС. `cancelLoad()` отменяет обе и останавливает watcher.

### TabsModel — только активная вкладка

Требование скорости: **слежение живёт ровно у одной вкладки — активной**. Десять вкладок на SMB означали бы десять потоков FSEvents, каждый со своим соединением к серверу.

Фоновая вкладка вместо слежения помечается устаревшей: при уходе с вкладки её watcher останавливается, при возврате — папка перечитывается, если пока её не было видно что-то менялось. Узнать это без слежения нельзя, поэтому правило простое и дешёвое: **возврат на вкладку всегда запускает фоновое обновление** тем же `applyExternalChange`, с `hasModifications: true`.

Это одно чтение имён на переключение вкладки — 0.7 мс на локальной папке. Показ при этом мгновенный: список уже есть, обновление приходит поверх. Смысл вкладок («возврат без ожидания») сохраняется, `hasLoaded` и ленивая первая загрузка не меняются.

Изменения в `TabsModel`: приватный `updateWatching()` — останавливает слежение у всех вкладок, кроме активной, и запускает у активной; зовётся из всех веток, меняющих `activeIndex` (`select`, `close`, `closeOthers`, `closeToTheRight`, `open` при `activate: true`). Тот же список мест, что уже обязан звать `loadIfNeeded`.

`TabsModel` и `TabState` получают параметр инициализатора `makeWatcher: () -> any DirectoryWatching` со значением по умолчанию — каждая вкладка создаёт свой watcher, а тестам нужно подставить фейк во все вкладки сразу. Фабрика, а не готовый экземпляр: watcher у каждой вкладки собственный, общий вёл бы к тому, что переключение вкладок переставляло бы слежение и у фоновых.

### Жизненный цикл окна

Слежение останавливается, когда окно неактивно: `NSApplication.didResignActiveNotification` → `stop()`, `didBecomeActiveNotification` → `start()` плюс однократное обновление. Пользователь, ушедший в другое приложение, список не видит, а FSEvents на сетевом томе продолжал бы держать соединение.

Подписка на нотификации живёт в `MainWindow` (`Sources/Pathway/`) и зовёт методы `AppState` — `NSApplication` в Core не тянем.

## Поток данных

**Внешнее создание файла.** Claude пишет `отчёт.docx` в открытую папку → FSEvents ждёт до 0.3 с, собирая пачку → колбэк на своей очереди фильтрует события по родительской папке → `MainActor` → `BrowserModel.applyExternalChange(DirectoryChange(hasModifications: false))` → чтение имён в фоне → дифф находит одну новую запись → `items` обновляется, файл появляется в списке без метаданных → второй проход дочитывает размер и дату.

**Дозапись существующего файла.** FSEvents несёт `ItemModified` → `hasModifications: true` → состав списка не изменился, но у всех записей сброшен `metadataLoaded` → второй проход обновляет размеры и даты.

**Смена папки.** `navigate` → `reloadAsync` → в конце загрузки watcher переставляется на новую папку. Прежняя перестаёт отслеживаться тем же вызовом `start`, без отдельного `stop`.

**Переключение вкладок.** `select(index:)` → `loadIfNeeded` (для непрочитанной) → `updateWatching()`: watcher прежней активной останавливается, у новой запускается и сразу зовётся обновление — папка могла измениться, пока вкладка была фоновой.

**Закрытие вкладки.** `cancelLoad()` отменяет обе задачи и останавливает watcher, после чего модель освобождается.

## Тесты

`Tests/PathwayCoreTests/DirectoryWatcherTests.swift`, свита «Слежение за папкой». Настоящий FSEvents не трогаем — тесты идут через `FakeDirectoryWatcher: DirectoryWatching`, который запоминает отслеживаемую папку, считает `startCount`/`stopCount` и умеет послать событие вручную.

Дифф и реакция модели:

- внешнее событие добавляет в список появившийся файл, **не перечитывая метаданные** уцелевших записей;
- внешнее событие убирает из списка исчезнувший файл;
- уцелевшие записи сохраняют прочитанные метаданные — `metadataLoaded` остаётся `true`, а не сбрасывается на каждом обновлении;
- событие с `hasModifications: true` сбрасывает `metadataLoaded` у всех записей;
- исчезнувший файл вычищается из `pane.selection`, а выделение уцелевших сохраняется;
- событие во время инлайн-переименования не меняет `items`;
- событие во время своей операции (`isBusy`) не меняет `items`;
- ошибка чтения при внешнем обновлении **не** выставляет `errorMessage` и не очищает список;
- событие, пришедшее после смены папки, не переписывает список новой папки (барьер `pane.path`).

Жизненный цикл слежения:

- `reloadAsync` запускает слежение за текущей папкой;
- переход в другую папку переставляет слежение, **а не заводит второе**;
- `cancelLoad()` останавливает слежение.

`TabsModelTests` дополняется:

- слежение включено только у активной вкладки;
- переключение вкладок переносит слежение на новую активную;
- возврат на прочитанную вкладку запускает фоновое обновление, **не сбрасывая** `hasLoaded` и не показывая пустой список;
- закрытие вкладки останавливает её слежение.

Отдельный тест на настоящий `DirectoryWatcher` — один, под `.timeLimit`: создаёт временную папку, запускает слежение, создаёт в ней файл, ждёт события. Он проверяет, что параметры потока заданы верно; всё остальное поведение покрыто через `FakeDirectoryWatcher` без обращения к диску.

## Ручной чек-лист

- Открыть папку, создать в ней файл из Терминала (`touch`) — появляется сам, за полсекунды.
- Удалить файл из Терминала — исчезает из списка.
- Дописать существующий файл (`echo >>`) — размер в колонке обновляется.
- Создать 50 файлов одной командой — список обновляется одним махом, без мигания.
- Позиция скролла и выделение не сбиваются при обновлении.
- Начать переименование (F2), создать файл извне — редактор не закрывается, имя не теряется.
- Внешнее изменение во время архивации не сбивает прогресс.
- Переключиться на другую вкладку, создать файл в папке первой, вернуться — файл на месте.
- Уйти в другое приложение, создать файл, вернуться — файл появляется.
- Сетевая шара: создать файл с другой машины — появляется (или, если сервер событий не шлёт, обновляется по возврату в папку).
- Отмонтировать том с открытой папкой — алерта нет, список остаётся прежним.
- Папка с 5000 файлов: создание одного файла извне не вызывает заметной паузы.

## Вне рамок

- **Периодический опрос как замена FSEvents на сетевых томах.** SMB-сервер может не слать уведомления, и обход папки по таймеру — единственный способ это обойти. Он же — ровно та постоянная фоновая нагрузка, которой проект избегает. Пользователь в этом случае остаётся с обновлением по переходу в папку.
- **Слежение за фоновыми вкладками.** Десять потоков FSEvents на сетевом томе. Заменено обновлением при возврате на вкладку.
- **Рекурсивное слежение за поддеревом.** Нужно было бы дереву папок в сайдбаре, но там изменения видны при раскрытии узла, а рекурсия по домашней папке — это поток событий от каждого кэша каждого приложения.
- **Анимация появления и исчезновения строк.** `NSTableView` умеет `insertRows`/`removeRows`, но текущий `reloadIfContentChanged` работает полной перезагрузкой. Переход на построчные обновления — отдельная работа со своей ценой в сложности, и делать её вместе со слежением значит смешивать два изменения в одном.
- **Уведомление о внешнем удалении открытой папки.** Сейчас список просто перестаёт обновляться. Осмысленная реакция (перейти к родителю? показать заглушку?) — вопрос, который стоит решать отдельно, с ним же связано поведение при отмонтировании тома.
