# Самообновление через GitHub Releases

## Цель

Приложение установлено у нескольких коллег вручную из DMG. Выпуск новой версии сейчас означает
обход всех с просьбой скачать образ заново, и узнать, кто на какой версии сидит, невозможно.

Приложение должно само замечать вышедший релиз, показывать это в углу строки заголовка и по
нажатию обновляться целиком: скачать, подменить бандл, перезапуститься. Установка вне App Store,
круг пользователей узкий и доверенный.

## Почему это вообще возможно без Developer ID

Приложение подписано ad-hoc (`codesign --sign -`), платного аккаунта разработчика нет. Казалось бы,
Gatekeeper обязан заблокировать скачанную копию — но он проверяет подпись **только** у файлов с
атрибутом `com.apple.quarantine`.

Карантин ставит не система, а сама скачивающая программа: тот, кто объявил `LSFileQuarantineEnabled`
в своём Info.plist. Safari объявляет, `curl` — нет. В `Resources/Info.plist` этого ключа нет, значит
файлы, скачанные через `URLSession` внутри Проводника, карантина не получают. Проверка подписи не
запускается вовсе, и ad-hoc бандл стартует без единого вопроса.

Отсюда важное следствие: **ключ `LSFileQuarantineEnabled` в Info.plist добавлять нельзя.** Он сломает
обновление целиком, а связь неочевидна — ошибку будут искать в коде загрузки.

Ужесточения macOS Sequoia (правый клик → «Открыть» убран, нужен поход в Системные настройки)
касаются только первой установки, скачанной браузером. Обновлений они не затрагивают.

## Архитектура

### Core: `Sources/PathwayCore/UpdateService.swift`

Вся логика в Core, вью только показывает состояние и вызывает методы — граница проходит там же, где
у остальных сервисов.

**`AppVersion`** — разбор и сравнение версий. Сравнение почисленно по компонентам, а не строкой:
строковое дало бы `1.10.0 < 1.9.0`. Понимает префикс `v` в теге, лишние компоненты, мусор отбрасывает.

**`ReleaseInfo`** — версия, URL архива, размер, заметки к выпуску.

**`UpdateState`** — то, что видит UI: `.idle` / `.checking` / `.available(ReleaseInfo)` /
`.downloading(Double)` / `.readyToRestart` / `.failed(String)`.

**`UpdateService`** — проверка, скачивание, установка.

### Границы с системой — протоколами

Тот же приём, что у `Mounting`, `CredentialStoring`, `ShareBrowsing`, `TerminalRunning`. Без него
тесты полезли бы в сеть и в `/Applications`.

- **`ReleaseFetching`** — «дай последний релиз». Боевая реализация ходит в GitHub API, тестовая
  отдаёт заготовку и считает вызовы.
- **`UpdateInstalling`** — распаковка, проверка, подмена бандла. В тестах фиктивная: настоящая
  трогает установленную копию приложения.

### UI: `Sources/Pathway/UpdateBadgeView.swift`

Правый край строки заголовка, напротив названия. Четыре состояния:

| Состояние | Вид |
|---|---|
| Обычное | `1.0.0` тусклым мелким шрифтом — просто версия |
| Есть обновление | `↓ 1.1.0` акцентным цветом, кликабельно |
| Скачивается | Индикатор прогресса и процент |
| Скачано | Кнопка «Перезапустить» |

Клик по значку в обычном состоянии запускает проверку вручную. Заметки к выпуску — в поповере.

Пункт «Проверить обновления…» в меню «Проводник» рядом с «О программе», как принято в macOS.

## Поток данных

### Проверка

`GET https://api.github.com/repos/koklish/Pathway/releases/latest`, без авторизации: репозиторий
публичный, токен не нужен и в раздаваемом бинарнике был бы утечкой.

Из ответа берутся `tag_name` (вида `v1.1.0`), `body` (заметки) и URL первого ассета с расширением
`.zip`. Версия сравнивается с `CFBundleShortVersionString` работающего бандла.

Автоматически — при запуске и не чаще раза в сутки; `lastUpdateCheck` хранится в UserDefaults.
Лимит GitHub без авторизации — 60 запросов в час на адрес, при таком режиме недостижим даже если
все коллеги за одним NAT.

**Неудачная автопроверка молчит.** Нет сети, GitHub недоступен, ответ не разобрался — состояние
остаётся `.idle`, пользователь ничего не видит. Ошибка показывается только если проверку запросили
вручную: иначе приложение в дороге без интернета ругалось бы при каждом запуске.

### Установка

Приложение не может подменить себя на ходу: оно выполняется из этого бандла. Поэтому подмену делает
внешний скрипт, а приложение к тому моменту уже завершилось.

1. **Скачивание** — `URLSession` в `~/Library/Caches/com.pathway.filemanager/Updates/`. Прогресс в UI.
2. **Распаковка** — `ditto -x -k`, не Archive Utility: та распространяет карантин исходного архива на
   содержимое и портит симлинки внутри бандла.
3. **Проверка до подмены** — распакованное действительно бандл, `CFBundleIdentifier` совпадает,
   версия в его Info.plist новее текущей, `codesign --verify` проходит. Битый или чужой архив
   дальше этой точки не идёт: подменять бандл непроверенным — это ломать установленную копию.
4. **Скрипт-помощник** — приложение записывает его во временную папку, запускает отсоединённым и
   завершается само.
5. **Перезапуск** — новая копия стартует уже из подменённого бандла.

### Откат

Живёт в скрипте, а не в приложении: решение нужно принимать снаружи, когда приложение уже мертво
или падает на старте.

Скрипт ждёт завершения процесса, затем **переименовывает** текущий бандл в `Проводник.app.old`
(не удаляет — это и есть точка отката), ставит на его место новый и запускает. Если новая копия не
поднялась за 10 секунд — скрипт возвращает `.old` на место и запускает старую версию.

Новая версия при первом запуске находит `.old` рядом с собой и удаляет его: раз она выполняется,
запуск удался и откат больше не нужен.

Схема переживает и битую загрузку, и падение новой версии на старте.

## Выпуск релиза: `release.sh`

```
./release.sh 1.1.0
```

Скрипт проверяет чистоту рабочего дерева и отсутствие такого тега, проставляет версию в
`Resources/Info.plist`, собирает release-бандл во **временной** папке, пакует его `ditto -c -k`,
коммитит изменение версии, ставит тег `v1.1.0`, пушит и создаёт релиз через
`gh release create` с приложенным архивом. Заметки — из сообщений коммитов после прошлого тега.

Версия в Info.plist и тег релиза берутся из одного значения и разойтись не могут. Расхождение
означало бы, что коллеги видят предложение обновиться по кругу, уже стоя на новой версии.

Сборка идёт во временной папке, а не через `./build-app.sh`: тот закрывает запущенное приложение и
перезаписывает `/Applications/Проводник.app`. Выпуск релиза не должен трогать рабочую копию.

## Тесты

Только `PathwayCore`, как и весь остальной проект.

**`AppVersion`:**
- `1.10.0` больше `1.9.0` — почисленно, а не строкой
- равные версии не считаются обновлением
- префикс `v` в теге разбирается
- разное число компонентов: `1.2` против `1.2.0`
- мусор вместо версии не роняет разбор

**`UpdateService`** с фиктивным `ReleaseFetching`:
- релиз новее текущего → `.available`
- та же версия → `.idle`
- релиз старее → `.idle`
- ошибка сети при автопроверке → `.idle`, ошибка не показывается
- ошибка сети при ручной проверке → `.failed`
- релиз без ZIP-ассета → `.idle`
- повторная автопроверка в тот же день в сеть не идёт: счётчик вызовов в фиктивной реализации
  остаётся прежним (тот же приём, что `loadCount`/`existsCount` в `InMemoryCredentialStore`)
- ручная проверка ограничение суток игнорирует

UserDefaults изолируются приватным `makeDefaults()` со свежим suiteName на UUID, как в остальных
тестах.

## Ручной чек-лист

Подмену бандла и скрипт-помощник автотестами не покрыть, не рискуя установленной копией.

- [ ] Значок показывает текущую версию тусклым, пока обновлений нет
- [ ] После выхода релиза значок становится акцентным и кликабельным
- [ ] Наведение показывает заметки к выпуску
- [ ] Клик запускает скачивание, прогресс двигается
- [ ] По завершении появляется «Перезапустить»
- [ ] Перезапуск поднимает новую версию, значок снова тусклый и с новым номером
- [ ] Настройки, закладки и пароли серверов после обновления на месте
- [ ] `Проводник.app.old` удалён после успешного запуска
- [ ] Обновление без сети: молча ничего не происходит
- [ ] Ручная проверка без сети: показывается ошибка
- [ ] Битый архив (подложить руками) не портит установленную копию
- [ ] Откат: заведомо нерабочая новая версия возвращает старую
- [ ] Обновление под обычным пользователем (не администратором) — система просит пароль

## Попутные правки

- **Версия `0.1` → `1.0.0`.** Первый релиз задаёт версию по-настоящему; приложением уже пользуются.
- **Заголовок окна.** `PathwayApp.swift:9` — `Window("Pathway", id: "main")`, хотя пользователю везде
  видно «Проводник». Правка касается той же строки заголовка, куда встраивается значок версии.

## Вне рамок

- **Дельта-обновления.** Бандл весит единицы мегабайт, экономия не стоит усложнения.
- **Каналы beta/stable.** Круг пользователей — несколько человек, разделять некого.
- **Откат на произвольную версию.** Откат на предыдущую покрывает реальный случай (новая не
  запустилась); выбор из истории версий — интерфейс ради интерфейса.
- **Тихая установка без спроса.** Пользователь должен понимать, почему приложение перезапустилось.
- **EdDSA-подпись релизов.** Сейчас доверие держится на HTTPS до api.github.com и контроле над
  аккаунтом. Захват аккаунта означает раздачу произвольного кода коллегам — принято сознательно
  для узкого круга. Место в архитектуре оставлено: `ReleaseInfo` дополняется полем подписи, проверка
  встраивается в `UpdateInstalling` перед подменой, остальное не меняется.
- **Уведомления в Центре уведомлений.** Значка в заголовке достаточно, разрешение спрашивать не за что.
