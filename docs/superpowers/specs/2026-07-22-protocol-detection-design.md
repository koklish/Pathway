# Спека: автоопределение протокола и подключение по FTP

Дата: 2026-07-22

## Цель

Подключение к серверам произвольного типа без выбора типа пользователем.

Сейчас подключение работает только по SMB: `ServerAddress.parse` молча подставляет схему `smb` для любого адреса без схемы, а диалог на пустой шаре всегда идёт в `smbutil view`. Адрес вида `31.31.196.75` из панели хостинга (логин, пароль, голый IP — и всё) подключить нельзя.

Итог работы:
- Голый хост или IP подключается сам: приложение определяет протокол пробой портов.
- FTP-серверы монтируются и открываются на чтение.
- На томе только для чтения команды записи погашены заранее, а не падают с ошибкой после подтверждения.

Явно принятое ограничение: **NetFS монтирует FTP только read-only.** Запись по FTP потребовала бы собственного FTP-клиента и виртуальной файловой системы — это отдельный проект, здесь его нет.

## Архитектура

### 1. Проба портов (Sources/PathwayCore/ProtocolProbe.swift)

Новая граница с ОС, оформленная тем же приёмом, что `Mounting` и `CredentialStoring`:

```swift
public protocol PortProbing: Sendable {
    func isOpen(host: String, port: UInt16, timeout: TimeInterval) async -> Bool
}

public struct ProtocolProbe: Sendable {
    public init(prober: any PortProbing = SocketProber())
    public func detect(host: String) async -> String?   // "smb" / "afp" / "ftp" / nil
}
```

`SocketProber` — неблокирующий `connect(2)` плюс `poll` с таймаутом 1 с. Не `URLSession` и не `Network.framework`: нужен факт «порт принимает соединение», а не рукопожатие протокола, и `connect` даёт это дешевле всего.

`detect` пробует 445 (SMB), 548 (AFP), 21 (FTP) **параллельно** через `withTaskGroup`, дожидается всех и возвращает первый по фиксированному приоритету SMB → AFP → FTP среди открытых. Не первый ответивший: иначе результат зависел бы от скорости сети и на хосте с несколькими открытыми портами менялся бы от запуска к запуску.

Таймаут 1 с на все три пробы сразу, поскольку они идут параллельно.

`PortProbing` протоколом — чтобы `StubPortProber` в тестах отдавал заданный набор открытых портов, не касаясь сети.

### 2. Схема становится опциональной (Sources/PathwayCore/ServerAddress.swift)

Сейчас `parse` подставляет `smb` для адреса без схемы (`defaultScheme`, строка 24). Это ровно то место, где сегодня теряется различие между «пользователь сказал SMB» и «мы не знаем».

```swift
public struct ServerAddress {
    public let scheme: String?      // было: String
    public let host: String
    public let share: String
}
```

`defaultScheme` удаляется. Решение о протоколе принимает уровень выше — `ConnectServerModel`.

Схема известна без пробы в трёх случаях, и все три сохраняют приоритет над ней:
- явный префикс: `smb://`, `ftp://`, `afp://`;
- UNC-синтаксис `\\host\share` — он однозначно означает SMB;
- закладка: `ServerBookmark.address` уже хранит канонический адрес со схемой.

Это единственное ломающее изменение в существующем типе. Провести его нужно по всем вызовам: `PathInput.parseNetwork`, `MountedServers.scanNetworkMounts` (там схема всегда приходит из `f_mntfromname`, то есть известна), `ServerConnection`, `NetworkPath.unc`, плюс существующие тесты.

`var url` при `scheme == nil` возвращает `nil` — адрес без протокола смонтировать нельзя, и тип это выражает.

### 3. Диалог подключения (Sources/PathwayCore/ConnectServerModel.swift)

Новый шаг в `Step`: `.detecting`. Пользователь нажал «Подключиться» на адресе без схемы → нейтральный спиннер «Подключение…» → `ProtocolProbe.detect` → маршрутизация:

| Определено | Следующий шаг | Режим входа |
|---|---|---|
| SMB, шара пуста | `.shares(host:)` | Гость |
| SMB, шара задана | `.credentials` | Гость |
| AFP | `.credentials` | Гость |
| FTP | `.credentials` | Зарегистрированный |
| ничего не открыто | `.address` + ошибка «Сервер не отвечает» | — |

Текст на шаге `.detecting` намеренно нейтральный: о протоколах пользователю не сообщается вообще — ни в процессе, ни после. Тип виден только косвенно, по адресу `ftp://31.31.196.75` в сайдбаре.

**`.shares` — ветка исключительно SMB.** `smbutil view` о других протоколах не знает; у FTP и AFP понятия шары нет, там корень `/` и путь на шаг короче.

**Режим входа «Зарегистрированный» для FTP — не косметика.** Анонимный вход на FTP почти всегда закрыт, и предложенный по умолчанию «Гость» дал бы гарантированный провал первой попытки с невнятной системной ошибкой. Формально `anonymous` через NetFS работает, поэтому вариант «Гость» остаётся доступным — он просто не предлагается.

Кнопка «Показать папки» (`ConnectServerView.swift:303`) появляется теперь только когда протокол уже известен как SMB и шара пуста. На адресе без схемы кнопка всегда «Подключиться»: до пробы неизвестно, есть ли у сервера понятие папок.

Подключение к закладке пробу **пропускает** — схема в `ServerBookmark.address` уже есть, лишняя секунда ожидания на каждом повторном подключении не нужна.

### 4. Монтирование (Sources/PathwayCore/ServerMounter.swift)

`mount` уже строит URL из схемы `ServerAddress` и вызывает `NetFSMountURLSync` — для FTP менять нечего.

Одна правка в `MountError.describe(code:host:)`: отказ аутентификации на FTP приходит другим кодом, чем на SMB, и без явной ветки превратился бы в невнятное «ошибка -1». Текст ошибки, как везде в проекте, рождается в Core.

Keychain работает без изменений: `kSecAttrProtocol` уже маппится из схемы (`ServerCredentials.swift:107`), для `ftp` есть `kSecAttrProtocolFTP`.

`MountedServers`, `ServerBookmarks` и дедупликация тоже без изменений — идентичностью везде служит `url.absoluteString`, и схема уже её часть.

### 5. Read-only том (BrowserModel, Commands.swift)

```swift
public private(set) var isReadOnlyVolume: Bool
```

Вычисляется в `BrowserModel` при загрузке каталога, на третьей ступени (метаданные) — там уже читаются ресурсные ключи.

Источник — `URLResourceKey.volumeIsReadOnlyKey` текущей папки, **а не** проверка `scheme == "ftp"`. Том бывает только для чтения по любой причине: образ диска, права, заблокированная флешка. Правильный ответ даёт файловая система, догадки о протоколе тут лишние и неполны.

В `CommandRegistry` условие `isEnabled` расширяется для команд, которые пишут на диск: `newFolder`, `rename`, `moveToTrash`, `paste`, `compress`. Архивация входит в список, потому что архив создаётся рядом с исходником.

Реестр — единый источник правды, поэтому главное и контекстное меню гаснут вместе, без дублирования условия.

Копирование **из** тома (`copy`, `open`, `revealInFinder`) остаётся доступным: это чтение, и ровно в нём смысл сценария.

## Поток данных

1. Пользователь вводит `31.31.196.75`, нажимает «Подключиться».
2. `ServerAddress.parse` возвращает адрес со `scheme == nil`.
3. `ConnectServerModel` переходит в `.detecting`, вызывает `ProtocolProbe.detect`.
4. `SocketProber` параллельно пробует 445/548/21, возвращает `"ftp"`.
5. Модель переходит в `.credentials` с режимом входа «Зарегистрированный».
6. Пользователь вводит логин и пароль, `ServerConnection.connect` монтирует `ftp://31.31.196.75/`.
7. Закладка сохраняется с полным адресом; следующее подключение идёт мимо шагов 2–5.
8. `BrowserModel` при загрузке тома читает `volumeIsReadOnlyKey`, поднимает `isReadOnlyVolume`.
9. Команды записи в обоих меню недоступны.

- открыт только 21 → `detect` возвращает `"ftp"`;
- открыт только 445 → `"smb"`;
- открыты 445 и 21 → `"smb"` (приоритет, а не порядок ответа);

- не открыто ничего → `nil`;

- `ftp://31.31.196.75` даёт `scheme == "ftp"` и пустую шару;

### PathwayCoreTests/ConnectServerModelTests.swift (дополнения)

- SMB с пустой шарой ведёт в `.shares`;
- AFP ведёт в `.credentials` с режимом «Гость»;
- закрытые порты дают ошибку на шаге `.address`;

Проверяется установкой флага, без всякого монтирования.

## Ручной чек-лист

- `31.31.196.75` с логином и паролем от хостинга подключается и открывается.
- Ни на одном экране диалога слово «FTP» или «протокол» пользователю не показано.
- Повторное подключение к сохранённому FTP-серверу не показывает шаг «Подключение…» дольше обычного (проба пропущена).
- `\\samba.ip.pro` по-прежнему идёт по SMB и показывает список папок.
- Несуществующий адрес даёт «Сервер не отвечает» примерно за секунду, а не за таймаут монтирования.
- На смонтированном FTP-томе пункты «Удалить», «Переименовать», «Новая папка», «Вставить», «Архивировать» серые в обоих меню.
- Копирование файла с FTP-тома на локальный диск работает.
- Пароль от FTP сохраняется в Связке ключей и подставляется при следующем подключении.

## Вне рамок

- **Запись на FTP.** NetFS монтирует FTP только read-only; запись потребовала бы собственного FTP-клиента и виртуальной ФС поверх текущего браузера — отдельный проект.
- **SFTP.** Собственного монтирования в macOS нет, нужен сторонний FUSE-драйвер как внешняя зависимость.
- **WebDAV.** Технически монтируется через NetFS, но порт 80/443 открыт у любого веб-сервера, и проба портов дала бы ложное «это WebDAV» почти на каждом хосте. Требует другого способа определения, чем эта спека.
- **Индикатор «только чтение» в адресной строке.** Погашенные пункты меню — достаточный сигнал; отдельная работа по UI.
- **Настройка приоритета протоколов.** Порядок SMB → AFP → FTP зашит; пользовательской настройки нет.
- **Объединение двух парсеров адреса.** `ServerAddress.parse` и `PathInput.parseNetwork` останутся раздельными; сведение их — самостоятельный рефакторинг вне этой задачи.
