# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## О проекте

Файловый менеджер для macOS в духе Проводника Windows. Swift 6, SwiftUI + AppKit, SPM (не Xcode-проект). Внутреннее имя продукта — Pathway, пользователю везде видно «Проводник».

**Весь код, комментарии, тесты, спеки и сообщения коммитов — на русском.**

## Команды

```bash
swift build                       # сборка
swift test                        # все тесты (~255)
swift test --filter createsNewFolder          # один тест
swift test --filter FileOperationsTests       # один сьют
./build-app.sh [debug|release]    # собрать .app (debug по умолчанию)
./make-dmg.sh [release]           # собрать DMG
```

**`--filter` работает только по английским именам `func` и `struct`, не по русским строкам из `@Test("…")`/`@Suite("…")`.** Фильтр по русскому имени даёт «0 tests» **и нулевой код возврата** — прогон выглядит успешным, хотя не запустил ничего.

**`./build-app.sh` имеет побочные эффекты:** закрывает запущенное приложение через `osascript quit` и перезаписывает `/Applications/Проводник.app`. `./make-dmg.sh` вызывает его внутри себя, то есть делает то же самое. Чтобы проверить сборку, не трогая установленную копию, соберите бандл вручную во временной директории.

## Архитектура

### Граница PathwayCore / Pathway

Три таргета: `PathwayCore` (вся логика, без зависимостей), `Pathway` (executable, вью), `PathwayCoreTests`. Зависимости строго вниз: UI → Core.

**Граница проведена не по «AppKit нельзя», а по «нет типов, которые рисуют».** AppKit в Core импортируется в пяти файлах — но только ради системных сервисов: `NSWorkspace`, `NSPasteboard`, `NSAppleScript`. SwiftUI в Core нет нигде, и это следует держать так: он затащил бы вью-типы в тестовый таргет.

Отсюда два следствия, объясняющие непривычные места в коде:

- `Shortcut`/`ShortcutKey`/`ShortcutModifiers` в `Commands.swift` — собственные типы вместо `KeyEquivalent`. Мост в SwiftUI и AppKit живёт в `Sources/Pathway/ShortcutBridge.swift`. Иконка команды описана строкой (имя SF Symbol), `NSImage` строится тоже в мосте.
- `FavoritesStore.move(fromOffsets:toOffset:)` переписан вручную, чтобы не тянуть SwiftUI-расширение массива.

### Обратный канал «Core просит UI»

Команде вроде «Переименовать» нужен инлайн-редактор, «Архивировать» — диалог. Выполнить это внутри Core нельзя, поэтому команда **выставляет запрос**, а вью его исполняет и сбрасывает: `AppState.pendingRename`, `pendingCompress`, `pendingEditPath`, `BrowserModel.passwordRequest`.

Без этого приёма реестр команд пришлось бы разрезать пополам.

### Реестр команд

`Sources/PathwayCore/Commands.swift` — единственный источник правды для заголовка, иконки, шортката и доступности. Из него строятся главное меню (`AppCommands.swift`) и пункты контекстного меню (`FileListView.menuNeedsUpdate`).

Действие в контекстном меню **осознанно своё**: оно работает от `table.clickedRow`, а не от выделения — это нативное поведение macOS (правый клик по невыделенному файлу действует на него). Реестр даёт контекстному меню только оформление.

Гашение команд при вводе текста — гибрид: `⌘C`/`⌘X`/`⌘V`/`⌘A` перехватывает сам `NSTextField` через responder chain, а `F2`, `⌘⌫`, `⌘⇧N` гасит флаг `AppState.isEditingText`. Флаг поднимают три места: инлайн-переименование, адресная строка, модальные листы (сброс на `.onDisappear`, а не на кнопках — Esc закрывает лист мимо них).

Отсюда важное следствие: **буферные команды не проверяют `isEditingText` в `isEnabled`** — проверка стоит внутри `run`. Погашенный пункт меню перехватывает шорткат и никуда его не отдаёт, так что `!isEditingText` в `isEnabled` не давал бы `⌘C` дойти до `NSTextField`, и в полях диалогов копирование не работало бы вовсе. Живой пункт AppKit доставляет по responder chain: фокус в поле берёт текстовую операцию, фокус в списке — файловую. Проверка в `run` нужна потому, что responder chain стережёт только клавиши, а выбор пункта мышью приходит в `run` напрямую.

Шорткаты **не вешаются на кнопки** — по той же причине: привязанный к кнопке шорткат гаснет вместе с её `.disabled` и не работает при фокусе в списке файлов.

### Загрузка каталога: три ступени

`BrowserModel.reloadAsync` — кэш → имена → метаданные. Схема продиктована замерами на SMB: обход каталога стоит сотни миллисекунд, чтение метаданных поверх готового списка почти бесплатно.

- **Кэш** — синхронно, до первого `await`, чтобы возврат в виденную папку отрисовался в том же тике.
- **Имена** — `Task.detached(.userInitiated)`, `DirectoryLoader.loadNames` через `opendir/readdir` с типом из `d_type`. Не трогать диск ни разу принципиально: `URL` пошёл бы выяснять тип на диске — 4824 мс против 0.7 мс на 510 подпапках.
- **Метаданные** — `.utility`, ниже приоритетом. Только этот проход пишет в кэш: там лежат лишь полные записи.

Две защиты от гонок, и они **не дублируют друг друга**: `loadTask?.cancel()` при смене папки и проверка `pane.path == directory` после каждого `await`. Отмена кооперативна — уже запущенный `Task.detached` её не видит, поэтому нужен второй барьер.

`DirectoryCache` держит `showHidden` **частью ключа** (иначе ⌘⇧. отдал бы список противоположного режима), а инвалидация обходит обе вариации сразу. `copy(_:to:)` инвалидирует папку назначения, `move(_:to:)` — ещё и родителей источников.

### FileListView — обёртка над NSTableView

SwiftUI `List` не потянул тысячи строк, нативную сортировку по заголовкам, инлайн-редактор имени, контекстное меню и drag & drop. Внутри пришлось идти дальше: ручной `layout()` вместо Auto Layout (1.4 мс на ячейку → 56 мс на экран при скролле) и `IconCache` вместо `NSWorkspace.icon(forFile:)` (3.1 мс на строку → 125 мс на экран).

Три неочевидные вещи в `Coordinator` растут из одного корня — координатор создаётся один раз, а структура `FileListView` пересоздаётся на каждый рендер, и SwiftUI зовёт `updateNSView` на любое изменение модели, включая выделение:

- **`rebind(renamingItem:)`** — биндинг от первого рендера ведёт в отработавший экземпляр, запись в него не доходит до `@State`.
- **`reloadIfContentChanged`** — полный `reloadData` рвёт скролл. Подпись сравнения включает `metadataLoaded`, а не только URL: третья ступень загрузки меняет только метаданные при том же наборе URL, и сравнение по одним URL проглотило бы обновление.
- **`isSyncingSelection`** — `selectRowIndexes` синхронно порождает нотификацию, которая переписала бы модель тем, что таблица успела показать. Без флага `selectAll()` из меню схлопнулся бы обратно.

Похожие защиты от повторного входа: `editingItem` (повторный `makeFirstResponder` сбивал бы курсор посреди набора имени) и `isCancelling` (Escape тоже шлёт `controlTextDidEndEditing`).

### Ошибки

Два независимых канала — у них разные заголовки алертов:

- `BrowserModel.errorMessage` — файловые операции и архивы, всё сходится в приватном `describe(_:at:)`. Системные ошибки переводятся с **действием** в тексте: не «нет доступа», а «Выдайте приложению полный доступ к диску в Настройках → …».
- `FolderActions.errorMessage` — терминал и Claude, через `perform`; там `TerminalError` уже `LocalizedError` с готовым текстом.

`ArchiveError.passwordRequired`/`.wrongPassword` перехватываются **до** `describe` и превращаются в `passwordRequest` — состояние, а не текст: UI показывает лист ввода пароля. `CancellationError` глотается молча: отмена не ошибка.

Текст ошибки всегда рождается в Core, UI его только показывает.

### Сетевые серверы

`ServerConnection` собирает закладки, Связку ключей и NetFS так, чтобы UI получал один ответ: `.mounted` / `.needsCredentials` / `.failed`. `bookmarks` и `mounted` публичные (сайдбар рисует их напрямую), `credentials` и `mounter` приватные — интерфейс никогда не касается Keychain и NetFS сам.

Две вещи, которые легко сломать не заметив:

- **`exists` отделён от `load` намеренно**: чтение данных пароля заставляет macOS показать диалог доступа к Связке ключей, а факт наличия записи можно узнать без него. `InMemoryCredentialStore` в тестах считает `loadCount`/`existsCount` именно чтобы это стеречь.
- **`MountedServers` не персистится специально** — том мог отвалиться, пока приложение было закрыто. Взамен `adoptExistingMounts` подбирает тома через `getmntinfo`: диск могли смонтировать в Finder, и иначе он исчез бы из сайдбара совсем.

Публичного API для перечисления шар в NetFS нет, поэтому `ShareBrowser` запускает `smbutil view`, а разбор вынесен в чистую `ShareList.parse` (тестируется без сети).

### Процессы

Общего хелпера нет — три разные реализации, различия обоснованы:

- `ArchiveService.run` — async, прогресс по строкам вывода `-v`, отмена через `withTaskCancellationHandler` → `terminate()`. `try Task.checkCancellation()` стоит **перед** проверкой `terminationStatus`: убитый по SIGTERM процесс вернул бы ненулевой код и превратился бы в ложную `ArchiveError` вместо `CancellationError`.
- `ShareBrowser.run` — синхронный, вызывается уже внутри `Task.detached`.
- `TerminalLauncher` — «выстрелил и забыл»: терминал должен пережить Pathway.

`TerminalLauncher` открывает папку через `/usr/bin/open -a`, а не AppleScript: Launch Services отдаёт её терминалу без Apple Events, а значит без запроса разрешения на автоматизацию. Для запуска команды обхода нет — там AppleScript обязателен, и путь проходит двойное экранирование (shell внутри, AppleScript снаружи).

### Протокол на каждой границе с ОС

`Mounting`, `CredentialStoring`, `ShareBrowsing`, `TerminalRunning`, `PasteboardService(isolatedForTesting:)` — один приём, применённый ко всем границам. Благодаря ему Core тестируется без сети, Связки ключей и чужого буфера обмена. **Новую границу с системой стоит вводить так же.**

## Рабочий процесс

**Спека пишется до кода.** `docs/superpowers/specs/YYYY-MM-DD-<тема>-design.md`, структура: Цель → Архитектура (подразделы с указанием конкретных файлов) → Поток данных → Тесты → Ручной чек-лист → Вне рамок.

Раздел «Тесты» — не декларация, а список конкретных проверок, фактически ТЗ на тест-файл. Раздел «Вне рамок» перечисляет отвергнутое с причиной.

Паттерн подтверждён историей: спека → сервис → интеграция в модель → UI, каждый шаг отдельным коммитом.

### Тесты

swift-testing (не XCTest). Покрывается **только `PathwayCore`** — UI-тестов в проекте нет принципиально, вместо них ручной чек-лист в каждой спеке.

- `@Suite("Русское описание")` на `struct` с английским именем `<Что>Tests`; `@Test("русская фраза")` на функции с английским именем.
- Имена тестов часто содержат контраст, фиксирующий то, что легко сломать: «перемещает файл в Корзину, **а не** удаляет безвозвратно», «ключом служит полный адрес, **а не только** хост».
- `withTempDir` / `withTempDirAsync` объявлены на верхнем уровне в `DirectoryLoaderTests.swift` — он служит де-факто общим файлом хелперов.
- UserDefaults изолируются приватным `makeDefaults()` со свежим suiteName на UUID. Настоящая Связка ключей трогается только по opt-in: `.disabled(if: ProcessInfo.processInfo.environment["PATHWAY_KEYCHAIN_TESTS"] == nil)`.

**Осторожно с URL в тестах:** `DirectoryLoader` канонизирует пути (`/var` → `/private/var`), а склеенный вручную `dir.appendingPathComponent("file.txt")` — нет. Выделение и сравнения стройте от `model.items[…].url`, иначе `Set.contains` не найдёт совпадения.

### Комментарии

Объясняют **почему**, а не что. Типичная форма — двоеточие с обоснованием и явное «иначе …» / «а не …», описывающее отвергнутую альтернативу:

> «Не labelWithString: label не берёт фокус, и makeFirstResponder при переименовании молча возвращает false.»

Замеры производительности приводятся числами прямо в комментарии — они и есть обоснование архитектуры. Комментариев-пересказов кода в проекте нет; не добавляйте их. Секции размечаются `// MARK: - Русский заголовок`.

### Коммиты

Русский язык, без префиксов `feat:`/`fix:`. Заголовок — именная группа, не императив, часто с областью через двоеточие: `Core: DirectoryLoader и FileOperations с тестами`, `UI архивов: пункты меню, диалог архивации…`.

Тело подробное: 3–4 абзаца связного текста (не буллеты) — что сделано, почему именно так, что отвергнуто, какие ограничения остались. Хороший тон — назвать и то, чего изменение **не** чинит. Мелкие правки идут без тела: многословность пропорциональна нетривиальности.

Один смысловой шаг — один коммит.
